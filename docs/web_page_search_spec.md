# ウェブページ全文検索機能 - 詳細設計書

## 1. 概要
本機能は、ローカル環境のファイル検索に加え、指定したウェブサイトの URL 階層下にあるすべてのページをクロール・本文抽出し、FTS5 で一括全文検索を行うための拡張機能です。検索結果をクリックすると、キーワードがある位置へダイレクトにジャンプ（ハイライト）することができます。

---

## 2. アーキテクチャとデータ構造の拡張

既存の SQLite スキーマを活用して Web ページのデータも同一の全文検索（FTS5）へ統合します。

### 2.1 テーブルの拡張

#### `files` テーブル（既存スキーマへの列追加、または拡張）
Web ページとローカルファイルを区別するため、以下の列を追加・活用します。
- `source_type`: TEXT型 (デフォルト: `'local'`)。`local` または `web` を格納します。
- `normalized_path`: TEXT型。Web ページの場合は URL を正規化（小文字化、末尾スラッシュ統一、クエリパラメータ調整等）したものを一意のキーとして格納。

```sql
ALTER TABLE files ADD COLUMN source_type TEXT NOT NULL DEFAULT 'local';
```

#### `file_segments` テーブル（変更なし）
Web ページから抽出したテキストも、ローカルファイル同様に `file_segments` テーブルに分割（見出しごと、あるいは段落ごと）して保存します。
これにより、既存の `file_segments_fts` をそのまま利用した一括検索が可能です。

## 3. Web クローラーの設計仕様

### 3.1 クロール対象の制限・巡回ロジック
- **ベース URL**: ユーザーが指定した `http://example.com/docs/` 等の URL を開始点とします。
- **巡回範囲（スコープ）**:
  - ドメインの制限: 同一ホスト・同一ドメイン（例: `example.com`）のみ。
  - パス階層の制限: 開始点であるベース URL から始まるリンク（前方一致）のみ。
- **巡回方式**:
  - 幅優先探索 (BFS) または深さ優先探索 (DFS) による再帰走査。
  - 重複巡回を防ぐため、訪問済み URL セットを保持し判定。
  - 最大巡回階層（Depth: デフォルト 3）や最大ページ数の制限を設定可能とします。

### 3.2 本文抽出・構造化
- `requests` (または `httpx`) を用いて HTML を取得します。
- `BeautifulSoup4` や `trafilatura` 等のライブラリを用いて、ナビゲーション、ヘッダー、フッター等の不要な要素を除去し、メインコンテンツのみを抽出します。
- タイトルタグ (`<title>`) または見出しタグ (`<h1>`) を `files.file_name` に設定します。

### 3.3 インデックス作成のパフォーマンス配慮
- 取得した各ページに対して、100件ごとにバッチで DB コミットを行い、I/O 負荷を削減します。
- クロール時は接続タイムアウト・エラーハンドリング（404, 500 等）を実装し、エラーとなったページは `failed_files` テーブルに記録します。

## 4. 検索とキーワードジャンプの仕様

### 4.1 検索の対象制限とパラメータ制御
- **影響範囲の極小化**:
  - 既存の `/api/search` への影響をゼロにするため、検索APIにオプトインパラメータ `source_type` (デフォルト: `local`) を導入します。
  - **API経由（デフォルト）での動作**: パラメータを指定しない場合、`source_type='local'` として処理され、Webページのインデックス（`source_type='web'`) は一切検索されません。これにより、既存のランチャーや他アプリからのAPI利用で結果が煩雑になることを防ぎます。
- **GUI限定のWebページ検索**:
  - Web UIフロントエンドからの検索時のみ、GUI上のトグル・タブの状態に応じて `source_type=web` パラメータをAPIへ送信し、Webページの全文検索を行います。
  - Web検索用のクエリでは、`files.source_type = 'web'` に絞り込んだ上で、FTS5 を用いて該当のセグメントを抽出します。

### 4.2 キーワード位置へのジャンプ
検索結果から Web ページへのリンクをクリックした際、該当キーワードへ直接飛ぶために **URL Text Fragments** (`#:~:text=...`) を利用します。

#### 生成ロジック
1. ヒースト（Hit）した検索キーワード（例: `react`）を取得。
2. クリック時に生成されるリンク URL の末尾に `#:~:text=キーワード` を付与します。
   - 例: `http://example.com/docs/intro#:~:text=react`
3. 複数単語の場合や曖昧さを排除したい場合は、ヒットしたセグメントのコンテキスト（前後のテキスト）を用いて、`#:~:text=start_text,end_text` の形式で URL を構築します。

これにより、Google Chrome などの対応ブラウザで開いた際、該当箇所が自動的にスクロール・ハイライト表示されます。

## 5. UI/UX 仕様

- **検索対象の切り替え**:
  - 検索バーの直下または横に、**「ローカル検索」と「ウェブページ検索」を切り替えるタブまたはトグルボタン**を配置。
  - ユーザーが選択したモードに応じて、フロントエンドがAPIへ渡す `source_type` パラメータを自動で切り替えます。
  - 切り替えによってローカルファイルとWebページの結果が混ざるのを完全に防ぎ、画面の視認性を高く保ちます。
- **登録フォーム**:
  - 検索バーのフォルダ入力欄は、検索対象モードが「Web」のときベース URL 入力欄として扱う。
  - ユーザーが `https://example.com/docs/` のようなトップページ URL を入力して「検索対象に追加」を押すと、フォルダ追加と同じ `targets` 管理へ `source_type='web'` として保存し、指定階層下のページをクロールしてインデックスする。
  - 入力項目: ベース URL, 巡回深度（既存の階層入力を使用）。除外キーワードはローカル用とは別のWeb専用除外キーワードを適用する。
  - 既存の「検索対象フォルダ」一覧には Web URL も同じ一覧に表示し、有効/無効・再インデックス・削除を同じ操作体系で扱う。
- **インデックス済みWebページ管理**:
  - 画面上部の `インデックス済みWebページ` タブから、`files.source_type='web'` のページをURL単位で一覧表示する。
  - 各カードにはURL、ページ数、最終取得日時を表示し、クリック操作で対象URLをブラウザの新規タブで開く。
  - 絞り込み、全選択、選択したページのインデックス削除はインデックス済みフォルダと同じ操作体系で扱う。
- **Web除外キーワード**:
  - ハンバーガーメニューの設定に `Web除外キーワード` を置き、ローカルフォルダ用の `除外キーワード` と分離して保存する。
  - WebクロールとWeb検索では `web_exclude_keywords.txt` の値を既定の除外条件として使う。
- **検索結果リスト**:
  - Web ページである検索結果には、ローカルファイルと区別がつくよう地球アイコン（`globe`）やバッジ（`Web`）を表示。
  - クリックするとブラウザでその URL に直接アクセスし、Text Fragments によって目的の箇所へ自動ジャンプします。
- **デザイン**:
  - 既存の Glassmorphism や、HSL ブルー (`hsl(217, 91%, 60%)`)、ダークモードテーマ (`#0f172a`) を一貫して適用。
