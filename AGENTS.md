# Local Fulltext Search - 開発仕様書

## プロジェクトの目的
ローカル環境にある文書・設定・図表ファイル(`.md`, `.json`, `.txt`, `.xml`, `.pdf`, Office, `.msg`, `.excalidraw`, `.dio`)を高速に全文検索し、画像・音声など本文抽出しないファイルはファイル名検索で補助するためのアプリケーション。

## 開発ルール

### Test-Driven Development (TDD)
- 原則としてテスト駆動開発 (TDD) で進める。
- 期待される入出力に基づき、まずテストを作成し、テストの失敗を確認してから実装を行う。

### コメント・仕様の記述
- 各ファイルの冒頭や主要な関数の前には、日本語のコメントで仕様を明確に記述すること。
- （例）
  ```typescript
  /**
   * 2点間のユークリッド距離を計算する
   **/
  ```

### ドキュメントについて
- コアな仕様はこの `AGENTS.md` に簡潔にまとめる。
- 新規のドキュメントは `docs/` フォルダの中に作成し、コードの変更に合わせて常に最新化する。
- 構成図などが必要な場合は、Excalidraw 形式で作成・更新する。

## UI・デザインの仕様

### ダークモード & モダンデザイン
- **テーマ**: ダークモード (`#0f172a` ベースのネイビー/ブルー系)
- **Glassmorphism**: 主要コンポーネント（カードやサイドバー等）の背景半透過 + `backdrop-filter: blur(16px)` を利用して、背後のオブジェクトが透ける美しい表現を採用する。
- **アクセント**: 視認性の高いフューチャリスティックなブルー（`hsl(217, 91%, 60%)`）をプライマリカラーとし、洗練された印象を与える。
- **インタラクション**: 各ボタンやカードがホバー時になめらかに浮き上がる (translateY) 等、動的で触り心地の良いマイクロアニメーションを取り入れる。
- **タイポグラフィ**: `Inter` などのクリーンな Sans-Serif フォントを使用。

## パフォーマンス最適化

### インデックス作成
- **バッチ commit**: ファイルごとの commit を廃止し、100件ごとのバッチcommitで I/O 削減。
- **os.scandir 走査**: `rglob` の代わりに `os.scandir` で再帰走査。除外ディレクトリの子ディレクトリに入らない。
- **エントリ名のみ除外チェック**: 再帰走査で親パーツはチェック済みなので、`entry.name` だけで除外判定。
- **除外キーワード**: `frozenset` による `O(1)` ルックアップ。非ASCII キーワードは事前分離。
- **IN 句バッチ削除**: 削除済みファイルのレコード削除を IN 句でバッチ化。`file_segments` を先に DELETE し FTS5 トリガーを発火。
- **failed_file 一括クリア**: ファイルごとの DELETE を廃止し、インデックス完了後に一括クリーンアップ。
- **ワーカー数**: I/Oバウンド対応で `min(16, cpu_count * 2)` に設定。

### SQLite 設定
- WAL モード + `synchronous = NORMAL` + `cache_size = -64000` + `temp_store = MEMORY` + `mmap_size = 256MB`
- **範囲クエリ**: `LIKE` の代わりに、子孫パス用の共通境界ヘルパーで `>= / <` を組み立てて B-tree インデックスを活用。`/` や `C:/` のようなルートパスでも壊さない。
- 詳細は `docs/performance.md` 参照。

### 検索
- **CTE 統合**: FTS5 の COUNT と結果取得を1クエリに統合。
- **正規表現イテレータ**: `fetchall()` の代わりにカーソルイテレータでメモリ効率化。
- **デフォルト並び替え**: 初期状態は `default` とし、ファイル名・Obsidian `title` / `aliases` の一致レベル、`top` タグ、同じ検索語からのクリック履歴、本文関連度バケット、利用価値スコア、更新日時の順で優先する。検索語別クリックは30日基準で減衰する。利用価値は飽和させたアクセス数55%、Obsidian重要度25%、90日基準の更新鮮度20%で合成する。作成日・更新日・アクセス数の個別並び替えではこれらを強制優先しない。
- **最大走査階層（index_depth）の無制限化およびUIコンパクト化**: UIでの階層指定が未入力（空欄または `None`）のときは、上限なし（無限大、内部的には 99999 階層）として走査する。また、検索画面の縦幅を削減して結果一覧の視認性を高めるため、階層入力UIを1段目（フォルダ指定行の右側）へインライン配置し、説明テキストはホバー時のみツールチップ（`title` 属性）で表示する。

### DB 接続
- 起動時に `lifespan` コンテキストマネージャでスキーマを初期化する。
- API リクエストごとに SQLite 接続を開閉し、スレッド間で同一接続を共有しない。
- インデックス中止要求は DB の `index_runs.cancel_requested` にも保存し、別プロセスのスケジューラー実行から検知できるようにする。

## デスクトップランチャー (Flet / PyObjC)
- **フォルダ**: `launcher/` にソースコードとドキュメントを分離して管理。
- **UI**: macOS では PyObjC / Cocoa `NSPanel`。その他 OS では Flet をフォールバック使用。
- **ホットキー**: `Command + Option` (Mac) / `Windows + Alt` (Windows Flet) / `Windows + Alt + Space` (Windows WPF) によるグローバル表示切り替え。
- **設定**: `LAUNCHER_API_BASE_URL`（Search/API接続先、既定 `http://127.0.0.1:8079`）、`LAUNCHER_WEB_BASE_URL`（Open/UIハブ、既定 `http://127.0.0.1:8001`）、`LAUNCHER_SEARCH_LIMIT`、`LAUNCHER_REQUEST_TIMEOUT` を環境変数で設定可能。
- **挙動**: 
  - macOS 版はマウスカーソルのあるディスプレイに表示。Flet 版は Flet の `window.center()` で中央表示。
  - フォーカス喪失時（blur）に自動非表示。
  - **検索挙動**: 
    - macOS: 検索時に登録済みフォルダのインデックス更新を許可。
    - その他 OS: 既存インデックスのみを使用して高速にレスポンス。
  - 立ち上がった（再表示された）時に、検索欄に残っているテキストを全選択状態にしてフォーカスする。
  - Web版のデザイン（Glassmorphism）を踏襲。
  - ウィンドウの表示・非表示時に、キーボードの押下状態（pynputフック）をリセットして、Enterキーが反応しなくなるのを防止する。
  - **Windows Enter/IME ガード**: Windows の検索欄フォーカス中の単独 Enter は IME 変換確定として扱い、結果起動には使わない。ただし上下矢印キーで結果を選択した直後の Enter は結果起動として扱い（`_arrow_navigated` フラグ）、テキスト入力でフラグをリセットする。
  - **gantt 送信 UI**:
    - gantt タスク追加用のメモ画面を、「タスク名（名前）」と「メモ」の独立した入力ボックスに分離。
    - 「送信」および「キャンセル」ボタンを追加。
    - メモ画面内では `Tab` / `Shift + Tab` によって「名前欄 ⇔ メモ欄 ⇔ 送信ボタン ⇔ キャンセルボタン」の間でフォーカスを巡回移動させる。
    - メモ入力欄およびタスク名入力欄での `Return` / `Enter` （通常のEnterまたはShift+Enter）はすべて改行（または通常の入力継続）とし、即時送信は行わない。
    - 送信処理は、「送信」ボタンをクリックしたとき、または「送信」ボタンにフォーカスがある状態で `Return` / `Enter` キーを押したときのみ実行する。
    - メモ画面での `Escape` キー押下時は、ランチャーを閉じるのではなく検索画面へ戻る挙動とする。
    - 送信成功時は入力値をクリアし、「タスク名」入力欄へフォーカスを戻してランチャーを開いたままにし、連続でタスクを追加可能にする。
    - 詳細は `docs/gantt_launcher_ui.md` 参照。

- **詳細は `launcher/docs/spec.md` 参照。**

## 外部Open/UIハブ
- **8001**: 別の必要アプリが提供する既存Openハブ。このリポジトリでは実装・起動・停止・監視・プロキシを行わない。
- **8079**: 検索・設定・click記録・`open-location`などのSearch/API面。primary openの入口にはしない。
- ランチャーとWebクライアントは外部8001の既存`/api/fullpath?path=...`と`/?path=...`へ従来どおり送る。8001停止時は接続エラーでよく、8079へフォールバックしない。
- DNS無し環境では`127.0.0.1`または固定IPだけを差し替える。詳細は`docs/open_hub.md`参照。

### Windows WPF ランチャー
- **フォルダ**: `launcher/windows/`。Windows x64 / .NET 8 WPFで、外部NuGet依存なし。
- **配布**: self-containedフォルダ版を第一候補とし、self-contained single-file版も発行する。会社PC側のSDK・ランタイム・インストーラーは不要。
- **Git配布**: 会社PCでビルドしないため、`launcher/windows/publish/folder` と `launcher/windows/publish/single-file` の発行成果物（EXE、.NETランタイム、同梱アセット）は C# ソース変更と同じコミットでGitへ追加・pushする。`bin/`・`obj/`、Python仮想環境、依存キャッシュは追跡しない。
- **ホットキー**: Windows標準APIによる `Shift` の2回押下（400ms以内、間に他キーを挟まない）で表示・非表示を切り替える。入力は遮断せず、`RegisterHotKey` は使用しない。
- **仮想デスクトップ**: 非表示時に検索ウィンドウを破棄し、次回表示時にHWNDを再生成して、呼び出した仮想デスクトップへ表示する。
- **フォールバック**: 発行済みEXEがない場合だけ既存Python/Flet版を起動する。
- **macOS版準拠**: WPF版も検索欄右側の拡張子フィルタ、gantt検索、タスク名・メモ・送信・キャンセル、共有parent、パスコピー、保存場所、フォルダ、ganttリンク、GUIボタン、上下キー循環選択を提供する。`Tab` / `Shift + Tab` は「検索欄 → 拡張子欄 → タスク名 → メモ → 送信 → 検索欄」を循環し、送信後のTabでは検索画面を表示する。`.py`、`.bat`、`.exe`、`.lnk` は8001を経由せず、親フォルダをcurrent directoryとして直接起動する。Windowsの検索性能仕様に従い、gantt未選択時は既存インデックスのみを検索する。
- **再表示**: 仮想デスクトップ移動のためUIウィンドウを再生成しても検索文字列は保持し、検索欄へ復元して全選択する。WPF初期版は半透明パネルを使用し、DWM/Acrylic背景ブラーは未実装。
- **WPF検索結果UI**: 横スクロールなしで結果全体を確認できる幅とし、長いパス・抜粋・操作ボタンは折り返して表示する。ランチャー表示直後は検索欄へフォーカスし、検索結果の先頭が選択済みなら矢印操作なしの `Return` / `Enter` でも選択結果を開く。
- **WPF検索対象**: ローカルファイルとWebページの両方を既存DBから横断検索する。WPFからの検索ではインデックス更新を行わない。

## ウェブページ全文検索
- **目的**: 指定したベースURL階層下の全ページをクロール・本文抽出し、ローカルファイルと同様に全文検索可能にする。
- **データ統合**: `files` テーブルに `source_type` 列（`local` or `web`）を追加。Webページのパス（URL）を `normalized_path` とし、本文を `file_segments` 経由で FTS5 へ登録する。
- **検索の切り替え・オプトイン**: GUI上に「ローカル検索」と「ウェブページ検索」を切り替えるトグル/タブを設置。検索APIでは `source_type` パラメータ（デフォルト: `local`）によって挙動を制御し、パラメータのないAPI呼び出し（デスクトップランチャー等）ではWebページ検索を完全に除外することで影響範囲を極小化する。
- **キーワードジャンプ**: 検索結果をクリックすると、URLの末尾に URL Text Fragments (`#:~:text=...`) を付与して開き、目的の箇所へ自動ハイライトジャンプする。
- **認証・JavaScript描画**: Web取得方式は通常HTTP（既定）・Microsoft Edge・Google Chromeから選択する。Edge/ChromeモードはPython版Playwrightで端末にインストール済みの正式版ブラウザを画面あり・アプリ専用永続プロファイルで起動し、Windows統合認証やJavaScript描画後のDOMを利用する。ブラウザ拡張機能およびPlaywrightブラウザの追加ダウンロードは不要とする。
- **詳細は `docs/web_page_search_spec.md` 参照。**

## gantt タスク検索
- **目的**: gantt アプリのタスク内容を、通常のローカルファイル検索とは分離した任意選択の検索対象として扱う。
- **API 連携**: 
  - 既定の取得先は `http://localhost:8000/api/tasks`。バックエンド設定 `SEARCH_APP_GANTT_API_BASE_URL` で変更可能。
  - レスポンスからタスク一覧を取得し、空白区切りの検索語を AND 条件で照合する。
- **検索仕様**:
  - `include_gantt_tasks: true` が指定されたときだけ gantt API を呼び出す（デフォルトは `false`）。
  - 検索結果は `source_type: "gantt"`, `full_path: "gantt://tasks/{id}"` として返す。
  - タイトルクリック時は `POST /tasks/{task_id}/open-input` を呼び出す。
  - タスク JSON に `link` / `url` / `href` / `input_url` / `external_url` がある場合のみ、結果アクションに「ganttのリンクを開く」を表示する。
- **UI**: Web 版検索バーに `gantt` の追加チェックを設置。ランチャー（macOS/Flet版）も同様に `gantt` チェックを有効にした時のみ結果に追加する。
- **詳細は `docs/gantt_task_search.md` 参照。**
